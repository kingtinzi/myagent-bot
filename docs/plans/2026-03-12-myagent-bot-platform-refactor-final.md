# MyAgent Bot 平台化重构方案（审查修订版）

## Summary
- 产品形态保持为“桌面客户端 + 外置设置页 + 平台后端 + 管理员后台”四层：桌面端负责注册登录与聊天，设置页负责用户配置/余额/充值，平台后端负责官方模型转发、计费、订单、协议留痕，管理员后台负责模型、接口、价格、用户、订单和审计管理。
- “官方模型”定义为你们自营的平台服务。用户选择“官方”后，请求统一走你们的平台后端，再路由到真实上游模型厂商；客户端不持有官方上游密钥。
- 账号登录使用 Supabase（邮箱密码），但钱包、支付、官方模型、价格规则、日志审计不放在 Supabase 里拼装，单独由 Go 平台服务承接。
- 首版采用“余额预付 + 分阶段支付上线”：
  - Phase 1：钱包、订单、人工确认入账/测试支付
  - Phase 2：微信/支付宝正式接入
- 合规要求前置：注册必须同意《用户协议》《隐私政策》；充值前必须再次确认《充值服务协议/官方模型服务协议》，并显著展示金额、资金用途、计费规则、退款规则、主体信息、争议处理方式。

## Key Changes
- **应用账号与登录入口**
  - 桌面应用启动先显示注册/登录弹窗；未登录不进入聊天界面。
  - 应用账号与现有 OpenAI/Anthropic/Google provider auth 完全分离；设置页中统一称后者为“第三方模型授权”。
  - 登录成功后，平台后端返回用户基本信息、官方模型可用状态、余额摘要、协议版本状态。

- **官方模型平台代理**
  - 新增平台代理聊天入口 `POST /chat/official`，由平台后端决定上游 provider、上游 model、API base 和回退策略。
  - 管理员后台维护：
    - 官方模型展示信息
    - 上游 provider / model id / API base
    - 启用状态
    - 默认路由与回退路由
    - 价格规则版本
    - 生效时间
  - 客户端只消费平台下发的“可售官方模型目录”，不接触上游商业机密配置。

- **钱包、充值、计费**
  - 用户在外置设置页查看余额、充值入口、交易流水、消费记录、当前官方模型价格说明。
  - 首版计费规则锁定为：
    - 以“上游返回的实际 token 用量”为主计费口径
    - 若某次调用拿不到可信 token 数据，则按该模型预设 fallback 单价计费
    - 计费规则和 fallback 规则必须展示在价格说明与服务协议中
  - 调用前进行余额校验；调用完成后写入消费记录并扣费；失败请求默认不扣费，若已预扣则自动冲正。
  - 钱包流水必须区分：充值、消费、退款、补偿、管理员调账、赠送（若后续启用）。

- **支付分阶段上线**
  - Phase 1：
    - 先实现充值订单、支付状态、人工审核入账/测试支付
    - 充值页文案、钱包、账本、协议确认、订单状态全部按正式方案做
  - Phase 2：
    - 接微信/支付宝正式支付
    - 接入支付回调、退款、对账、失败补偿
  - 这样避免在主体、法务、财务流程未完全就绪时直接上线真实预付余额支付。

- **管理员后台**
  - 单独 Web 管理后台，不复用用户设置页。
  - 至少包含：
    - 用户管理
    - 钱包/调账
    - 充值订单
    - 消费流水
    - 模型与上游路由
    - 价格规则版本
    - 协议版本
    - 风控开关
    - 审计日志
    - 投诉/侵权处理记录

## Compliance / Legal Guardrails
- **协议确认**
  - 注册/登录必须勾选《用户协议》《隐私政策》。
  - 充值前必须再次勾选《充值服务协议》《官方模型服务协议》。
  - 必须留存：用户 ID、协议版本、确认时间、客户端版本、IP/设备摘要。

- **重大条款显著提示**
  - 在充值确认页单独显著展示：
    - 充值金额
    - 到账余额类型
    - 资金用途：仅用于官方模型调用及相关平台服务费用
    - 计费方式与 fallback 规则
    - 退款规则
    - 主体名称、客服方式、争议解决方式
    - “余额不是银行存款/不产生利息”（若采用预付余额模式）
    - “模型输出可能存在错误或权利风险，商用前需自行复核”

- **上游准入审查清单**
  - 每个上游模型厂商接入前必须记录并审核：
    - 是否允许商业 API 集成
    - 是否允许终端用户在你们产品中使用
    - 是否允许平台代理/托管式服务/加价售卖
    - 是否允许输出用于商业用途
    - 数据使用与保留条款
    - 封号/停服触发条件
    - 侵权赔付/责任限制条款
  - 未通过审查的上游不得进入“官方模型”。

- **输入/输出权利与风险**
  - 用户协议中明确：用户保证其输入内容合法且有权使用。
  - 服务协议中明确：模型输出不保证准确、完整、无侵权风险，用户公开发布或商用前需自行复核。
  - 平台保留对违法、侵权、高风险内容进行限制、拒绝服务、封禁账号的权利。

- **隐私与数据治理**
  - 隐私政策中明确告知：
    - 用户输入、附件、日志、交易记录会发送到你们平台后端
    - 必要时再转发给上游模型厂商
    - 若涉及境外模型或跨境传输，单独说明接收方类别、处理目的、保存期限
  - 技术实现上强制最小化：
    - 钱包/支付数据与聊天内容分表
    - 管理员默认不可看原始聊天全文，只能看脱敏计费记录
    - 附件默认短期缓存，超期清理
    - 错误日志默认不落完整正文，必要时脱敏采样

- **退款与预付规则**
  - 方案需先锁定以下规则，不能留给实现者临场决定：
    - 余额是否永久有效
    - 未消费余额是否可退
    - 已消费部分不可退的定义
    - 赠送金额是否可退
    - 平台停服、模型下架、账号封禁时的退款处理
  - 首版建议默认：
    - 实付未消费余额可申请退款
    - 已消费金额不可退
    - 赠送金额不可退
    - 平台永久停服或长期不可用时，未消费实付余额可退

## Public Interfaces / Data Model
- **桌面端新增接口**
  - `GetAuthState() -> { authenticated, user, sessionExpiresAt }`
  - `SignUp(email, password)`, `SignIn(email, password)`, `SignOut()`
  - `GetOfficialAccessState() -> { enabled, balance, currency, lowBalance }`
  - `ListOfficialModels() -> [{ id, name, pricingVersion, enabled }]`
  - `OpenSettings()`, `GetBackendStatus()`

- **平台后端 API**
  - `POST /auth/signup`, `POST /auth/login`, `POST /auth/logout`, `GET /me`
  - `GET /agreements/current`, `POST /agreements/accept`
  - `GET /official/models`
  - `GET /wallet`, `GET /wallet/transactions`
  - `POST /wallet/orders`, `GET /wallet/orders/{id}`
  - `POST /wallet/refund-requests`
  - `POST /chat/official`
  - `POST /payments/wechat/callback`, `POST /payments/alipay/callback`
  - 管理端：
    - `/admin/users`
    - `/admin/models`
    - `/admin/model-routes`
    - `/admin/pricing-rules`
    - `/admin/orders`
    - `/admin/wallet-adjustments`
    - `/admin/agreement-versions`
    - `/admin/audit-logs`
    - `/admin/infringement-reports`

- **建议数据表**
  - `users`
  - `user_agreements`
  - `wallet_accounts`
  - `wallet_transactions`
  - `recharge_orders`
  - `refund_requests`
  - `official_models`
  - `official_model_routes`
  - `pricing_rules`
  - `chat_usage_records`
  - `admin_users`
  - `admin_audit_logs`
  - `infringement_reports`
  - `data_retention_policies`（可选）
- **版本化要求**
  - `pricing_rules` 必须支持版本号和生效时间
  - 协议文档必须支持版本号和生效时间
  - 每次充值、每次消费都要绑定当时生效的价格规则版本

## Test Plan
- **账号与协议**
  - 未勾选协议不可注册
  - 协议版本确认可追溯
  - 登录、登出、会话恢复、过期回登录页
- **充值与知情确认**
  - 未勾选充值协议不可支付/创建订单
  - 充值确认页正确显示金额、用途、计费方式、退款规则、主体信息
  - 人工入账/测试支付流程可闭环
  - 正式支付回调成功、失败、重复回调均幂等
- **计费与钱包**
  - 余额不足时禁止调用官方模型
  - 正常请求按 token 计费
  - 拿不到 token 时按 fallback 单价计费
  - 失败请求不扣费或自动冲正
  - 钱包流水、消费记录、退款记录、管理员调账均可审计
- **价格与协议版本**
  - 管理员改价后，仅新订单/新消费按新版本规则执行
  - 历史订单和历史消费仍能追溯旧版本
- **数据与权限**
  - 管理员默认不可直接查看原始聊天正文
  - 日志脱敏、附件过期清理、支付数据与聊天数据隔离正确
- **上游准入与模型发布**
  - 未通过合规审查的上游不能上架到官方模型
  - 管理员修改模型路由后，新请求立即生效
- **打包与交付**
  - Windows 输出安装包/便携包
  - macOS 输出 `.app` / `.dmg`
  - 客户端能连接不同环境的平台后端与设置页

## Assumptions
- “官方模型”是你们自营的平台服务，不是简单替用户转卖某个个人账号或消费版模型服务。
- 首版登录方式仅支持邮箱密码。
- 首版只做余额预付，不做订阅/月卡、优惠券、邀请返利、自动续费。
- 充值入口首版只放在外置设置页。
- 支付采用分阶段上线：先订单与钱包，后微信/支付宝正式接入。
- 当前方案是工程与产品实施计划，不替代正式法律意见；上线前仍需由律师基于你们选定的上游、收款主体、协议文本、支付路径、数据流向做最终合规审查。
