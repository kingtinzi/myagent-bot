# Launcher 扩展诊断页与一键修复交互稿（草案）

目标：在 Launcher 提供“扩展可用性一眼可见 + 一键修复”能力，降低用户手工命令成本。  
依赖后端：`/plugins/status`（M1/M2）与 `POST /plugins/repair`（M3）。

---

## 1. 页面结构

入口建议：

- 顶部 `设置` 页新增二级页签：`扩展诊断`
- 或聊天头部增加 `扩展健康` 按钮

页面分区：

1. **全局状态条**
   - 文案：`已检测 8 个扩展：5 就绪，2 受限，1 不可用`
2. **筛选与搜索**
   - 过滤：`全部 / 仅异常 / 仅可修复`
3. **扩展卡片列表**
   - 每卡片显示：状态、失败检查项、修复按钮
4. **操作日志面板**
   - 展示最近修复动作执行结果（成功/失败/耗时）

---

## 2. 扩展卡片信息层级

卡片字段（按优先级）：

- 扩展名 + `plugin_id`
- 状态徽标：
  - 绿色：`就绪`
  - 黄色：`可用但受限`
  - 红色：`不可用`
- 核心失败项（最多 2 条，超出折叠）：
  - 示例：`未找到 lobster 可执行文件`
  - 示例：`node_modules 缺失`
- 操作按钮：
  - `一键修复`
  - `查看详情`
  - `复制诊断`

---

## 3. 详情抽屉（Debug 友好）

信息块：

1. `检查清单`
   - 行级：`check_id` / `ok` / `detail` / `reason_code`
2. `解析顺序与命中来源`
   - 例如：`LOBSTER_BIN -> bundled -> PATH`
3. `推荐修复动作`
   - 显示 `repair_actions`（风险级别 + 是否需确认）
4. `技术信息`
   - `agent_id`、`extensions_dir`、`updated_at`

按钮：

- `执行修复`
- `导出 JSON`

---

## 4. 一键修复交互流程

### 4.1 无确认动作（low risk）

1. 点击 `一键修复`
2. 前端直接调用 `POST /plugins/repair`
3. 卡片进入 `修复中...`
4. 成功后自动刷新 `/plugins/status`
5. 结果 toast：
   - 成功：`修复成功，扩展已恢复就绪`
   - 失败：`修复失败，请查看详情日志`

### 4.2 需确认动作（medium/high risk）

确认弹窗：

- 标题：`确认执行修复动作`
- 正文：`将安装扩展运行时到应用私有目录，不修改系统全局 PATH。`
- 风险标识：`中风险`
- 按钮：`取消` / `继续`

二次确认后再发请求。

## 5. 错误态文案（可直接用）

### 5.1 常见错误

- `BIN_NOT_FOUND`：`未找到必需命令：{name}`
- `NODE_MODULES_MISSING`：`扩展依赖未安装（node_modules 缺失）`
- `NODE_TOO_OLD`：`Node 版本过低，需 >= {minVersion}`
- `CREDENTIAL_MISSING`：`扩展凭据未配置，无法执行相关能力`

### 5.2 CTA 文案

- 主按钮：`一键修复`
- 次按钮：`手动修复指引`
- 链接：`查看排查步骤`

## 6. API 对接草图

读取状态：

- `GET /plugins/status`
  - 使用字段：`runtime_status`, `checks`, `repair_actions`

执行修复：

- `POST /plugins/repair`
  - 请求：
    - `plugin_id`
    - `action_id`
    - `args`（可选）
  - 响应：
    - `ok`
    - `status`（`ok|needs_approval|failed`）
    - `result/logs/error`

轮询策略：

- 修复动作发起后，2s 轮询一次 `GET /plugins/status`
- 最多 30s；超时后提示用户手动刷新

---

## 7. 与“透明执行模式”的协同

建议在你已上线的“透明执行”旁增加扩展修复事件：

- `准备修复扩展 lobster`
- `执行 install_bundled_cli`
- `修复完成（耗时 2.1s）`

让用户在聊天界面也能看到系统做了什么，降低“黑盒感”。

---

## 8. 交互验收清单

- [ ] 卡片状态与后端 `runtime_status` 一致
- [ ] 点击修复后有 loading 态，不可重复提交
- [ ] 失败后能看到 `reason_code` 与日志
- [ ] 需确认动作一定弹窗，不可绕过
- [ ] 修复成功后 5 秒内状态更新
- [ ] 无后端新字段时页面优雅降级（隐藏修复按钮，显示“当前版本不支持自动修复”）
